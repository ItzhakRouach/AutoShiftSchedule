CRITICAL MISMATCH:
- m12_night physically covers 19:00-07:00 (touches both noon and night)
- But TWELVE_HOUR_FILLS[m12_night] = ['night'] ONLY
- TWELVE_HOUR_COVERS[m12_night] = ['noon', 'night']

This means:
1. For coverage counting: m12_night only counts toward NIGHT requirement
2. For availability/rest: m12_night must check both noon and night windows
3. For grid marking: commitTwelve() marks the grid based on rolesByShift, which is derived from planCover

## CRITICAL OBSERVATION #5: planCover() LOGIC

planCover() (twelve-plan.ts:59-89):
```typescript
for (const shift of fills) {  // fills = TWELVE_HOUR_FILLS[variant]
```

## KEY POTENTIAL BUGS

1. **ANCHOR DAY MISMATCH:**
   - openPairs() finds gaps on day D
   - tryVariant() picks a variant V for day D
   - planCover() fills the GAP on day D using FILLS[V]
   - commitTwelve() marks grid[D][shift] for each shift in rolesByShift
   
   Problem: What if m12_night is anchored on day D, but the gap is on a MORNING?
   - m12_night FILLS ['night'] only (fallback.ts:45)
   - morning is NOT in FILLS
   - But the shift gap might be on morning (next day?)
   
   Actually, openPairs processes the SAME day, so the gap must be on that day D.

2. **VARIANT PREFERENCE BYPASS:**
   - Phase 1 tries PREFERRED (day/night) for all gaps
   - Phase 2 tries non-PREFERRED (03-15/15-03) only if Phase 1 didn't close gaps
   
   But the loop is per-DAY, per-VARIANT. It tries ALL PREFERRED variants for all
   gaps on that day, looping. Only after PREFERRED has no more progress does it
   try non-PREFERRED.
   
   This should be correct, but the loop condition is subtle:
   ```typescript
   while (moved && openPairs(input, st, d.index).length > 0) {
     moved = false
     for (const v of PREFERRED) if (tryVariant(...)) moved = true
     if (!moved) for (const v of PREFERRED) if (tryDisplace(...)) { moved = true; break }
   }
   ```
   
   Once there's no more PREFERRED progress AND no displacement, the loop exits.
   Then Phase 2 runs.

3. **GRID RENDERING BUG (VISUAL):**
   - If m12_night is anchored on day D, it marks grid[D][night][role]
   - But the UI might render grid[D][night][role] in the NIGHT cell
   - If UI renders grid[D][morning][role] in a MORNING cell, and the 12h was
     meant to cover morning, that's a UI bug, not a scheduling bug.
   
   BUT: m12_night FILLS ['night'] only, NOT morning. So the morning gap remains.
   
4. **OVER-ASSIGNMENT ISSUE:**
   - Two employees both get m12_night on same (day, role)?
   - commitTwelve() does st.grid[day][shift][role].push(e.id)
   - If two different employees are committed as is12h variants for the same cell,
     both would be pushed to the array.
   - But the requirement check should prevent this: gap() returns max(0, need - have).
   - If need=1 and one employee already fills it, gap=0, so no second variant should
     be attempted.
   
   Unless... the first fill happens via an 8h, not a 12h? Then the second 12h
   variant can't also fill it (role conflict). But if BOTH are 12h... let me check.
   
   Actually, grid[day][shift][role].length includes both 8h and 12h. So after the
   first (whether 8h or 12h), gap becomes 0, and no second should be assigned.
   
   UNLESS two variants fire in PARALLEL (both sampled before either commits)?
   No, twelve-fill is serial: tryVariant commits before returning.

5. **REQUEST PREFERENCE ISSUE:**
   - A morning-requesting employee is available morning
   - openPairs(day) has a morning gap
   - Phase 1 tries m12_day (PREFERRED)
   - planCover() checks if m12_day FILLS morning → yes, m12_day fills morning+noon
   - orderedEmployees() prioritizes request-satisfied employees
   - But if an m12_3to15 was available that ALSO fills morning, the 12h pass
     doesn't consult request preference — it tries PREFERRED variants first, period.
   
   Actually, orderedEmployees() uses compareCandidates(), which DOES rank by
   request satisfaction. So the ordering should respect it.

6. **CRITICAL BUG: MORNING GAP ON DAY D, COVERED BY m12_night ANCHORED ON DAY D?**
   
   Let's trace:
   - Day D: morning gap exists, night is already filled
   - openPairs(D) = [{morning, roleId}]
   - Phase 1 tries m12_day → planCover() for m12_day
     - FILLS[m12_day] = ['morning', 'noon']
     - morning is in FILLS, and gap>0 → planCover() tries to fill it
     - If employee qualifies, plan[morning] = role, newGaps++
     - m12_day is tried first and succeeds
   - m12_night is never tried (alternative consumed by m12_day)
   
   So a morning gap should NOT be filled by m12_night on the same day.
   
   UNLESS... the only available employee is not allowed for morning but IS
   allowed for night?
   - morning gap, but employee not available morning
   - m12_day touched ['morning', 'noon'], availability blocks it
   - Phase 1 continues, tries nothing else (can't find eligible employee)
   - Phase 2 tries m12_night → but m12_night FILLS ['night'] only, not morning
   - m12_night cannot close a morning gap (not in FILLS)
   - Phase 2 tries m12_3to15 → FILLS ['night', 'morning'] → can fill morning
   - m12_3to15 is used instead
   
   This seems correct per preference order: if m12_day fails and m12_night can't
   address the morning gap, 03-15 is the last resort.

## HYPOTHESIS ON THE REPORTED BUG

Screenshots show:
(a) m12_night (19:00-07:00) rendered on MORNING cells
(b) Two employees both got 19:00-07:00 on same (day, role)
(c) Morning-requesting employees not placed on morning
(d) m12_3to15 / m12_15to3 used even though day/night pair should come first

Possible root causes:

1. **Rendering bug (not scheduling):**
   The schedule is correct, but the UI renders 12h assignments incorrectly.
   E.g., when displaying grid[day][shift][role] for morning, it shows
   employees with is12h=true even though they fill NIGHT (not morning).
   
   Clue: "m12_night assignments appear anchored on MORNING cells"
   - If the 12h canonical record says m12_night
   - But the UI renders it in the grid[day][morning] visual cell
   - That's a UI bug accessing grid[day][morning][role] instead of
     grid[day][night][role]
   
2. **FILLS vs COVERS confusion in UI:**
   The UI might use TWELVE_HOUR_COVERS (which is ['noon', 'night'] for m12_night)
   to decide which grid cells to show an employee.
   
   If it's rendering on BOTH noon and night for m12_night, that's correct visually.
   But if it's rendering on morning because of some mixup, that's the bug.

3. **Over-assignment from disabled deduplication:**
   commitTwelve() does NOT deduplicate by (day, shift, role).
   If two different 12h variants end up calling commitTwelve for the same
   (day, shift, role), both employees get pushed to grid[day][shift][role].
   
   But openPairs() should prevent this: after the first commit, gap becomes 0.
   Unless one commit removes an 8h that was filling the gap, then a second 12h
   can fill it? No, because commitTwelve updates grid immediately, so the next
   openPairs() call would see the gap as filled.
   
   UNLESS the displacement logic (tryDisplace) removes an 8h holder's shift
   BEFORE checking if the 12h's second gap is actually open?
   
   Looking at tryDisplace line 107:
   ```typescript
   const reusable = holderFree && openPairs(input, st, day).some((p) => p.shift !== shift)
   ```
   
   This checks if there's ANOTHER open gap AFTER displacing the holder.
   If yes, the 12h commit happens, and the holder is displaced.
   If no, the holder's shift is restored.
   
   So the displacement logic seems safe.

4. **CRITICAL ISSUE IN OPENAIRS / VARIANT SELECTION:**
   
   What if:
   - Day D has a morning gap + a night gap
   - Only ONE employee available for both (via 12h)
   - m12_day tries first (PREFERRED)
     - orderedEmployees() ranks candidates
     - tryVariant(m12_day) calls planCover()
     - Employee works 8h morning, so absorb exists
     - planCover absorbs morning, tries to fill noon → no noon gap → newGaps=0
     - planCover returns null (newGaps must be >0)
     - tryVariant fails, returns false
   - m12_night tries next (still PREFERRED)
     - tryVariant(m12_night) calls planCover()
     - Employee already committed morning → worksThatDay check blocks
     - tryVariant fails
   - displacement tries m12_day again with absorption
   - m12_night tries displacement → tries to absorb the morning shift,
     but m12_night FILLS ['night'] not ['morning'], so no absorption match
   
   Hmm, the logic checks:
   ```typescript
   const fills = TWELVE_HOUR_FILLS[variant]
   const base = sameDayBase(st, e, day)
   if (!base || !fills.includes(base.shift)) continue
   ```
   
   If m12_night FILLS ['night'] and the base shift is morning, this skips.
   So m12_night cannot absorb a morning 8h. Correct.

5. **MOST LIKELY: PLANCOVER CHOOSING WRONG ROLE FOR A SHIFT:**
   
   planCover picks the FIRST available role for each shift:
   ```typescript
   for (const role of Object.keys(roleReq)) {
     if (plan[shift]) break  // Stop after first
     if (!e.roleIds.includes(role)) continue
     if (gap(input, st, day, shift, role) > 0) {
       plan[shift] = role
       newGaps++
     }
   }
   ```
   
   If the input.requirements order is non-deterministic, this could pick
   different roles on different runs.
   
   Also, what if TWO roles are open for the same shift (e.g., morning needs
   2 מוקדן + 1 מאבטח = 3 people)? The code picks the FIRST available role
   the employee can fill. If employee only holds 1 role, that's fine.
   But if employee holds 2+ roles, it picks the first in Object.keys() order,
   which is insertion-order in modern JS.
   
   This might not be deterministic if the requirements are built dynamically.

6. **THE SMOKING GUN: FALLBACK.ts variantForShift vs twelve-fill.ts PREFERRED:**
   
   variantForShift() is used ONLY for suggestions (line 124 in buildTwelveHourSuggestions).
   
   But what if the code is wrong and variantForShift() is somehow used in auto-fill?
   Let me grep again...
   
   Actually, it's NOT. Only suggestions use it. Auto-fill uses the hardcoded
   PREFERRED order in twelve-fill.ts.

CONCLUSION:

The most likely bugs are:

1. **UI Rendering bug:** m12_night is correctly filled into grid[day][night]
   but rendered on grid[day][morning] cell (visual cell mismatch).

2. **Displacement edge case:** tryDisplace removes an 8h, then a 12h is committed,
   but the 8h is not actually freed (maybe reapplied incorrectly?).

3. **Order-dependent picking in planCover:** If the same-day gaps are
   processed in an order where the wrong gap gets a 12h first, causing
   the preferred gap to be uncovered.

4. **Cross-role absorption bug:** If an employee has multiple roles
   and the absorption logic picks the wrong role for a shift.
